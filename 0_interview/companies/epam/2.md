### TYPESCRIPT QUESTIONS

Q1: What is TypeScript for? • Adds static type checking to JavaScript • Catches errors at compile time (before runtime) • Better IDE support (autocomplete, IntelliSense, refactoring) • Makes code self-documenting and safer to refactor • Compiles to plain JavaScript

Q2: What is enum? • Named set of constants • Makes code more readable than magic strings/numbers • Two types: numeric enum (default, starts at 0) and string enum • Example: enum Status { Pending = 'PENDING', Success = 'SUCCESS' } • Modern alternative: union types with 'as const'

Q3: What is the difference between type and interface? • Both define object shapes and can be extended • Interface: uses 'extends', supports declaration merging, better for object shapes • Type: uses '&' intersection, supports unions/primitives/tuples, more flexible • Interface better for public APIs, Type better for unions and complex types • Consistency in your project matters more than the choice

Q4: What is <T>? • Generic type parameter - creates reusable type-safe code • Type is determined when used, not when defined • Like function parameters, but for types • Example: function identity<T>(value: T): T { return value; } • Used extensively in React: useState<T>, Array<T>, Promise<T>

Q5: What are utility types? Name and explain a few. • Built-in TypeScript helpers for type transformations • Pick<T, K>: Select specific properties from type • Omit<T, K>: Exclude specific properties from type • Partial<T>: Make all properties optional, Required<T>: make all required • Record<K, T>: Create object type with specific keys, ReturnType<T>: extract return type • Example: type UserUpdate = Partial<Pick<User, 'name' | 'email'>>

Q6: What is the difference between 'any', 'unknown', and 'never'? • any: disables type checking, avoid in production code • unknown: type-safe 'any', requires type checking before use • never: represents values that never occur (unreachable code, exhaustive checks) • unknown forces you to narrow type before using it • never useful for exhaustive switch cases and error handling

### REACT QUESTIONS

Q7: What's the virtual DOM? • JavaScript representation of real DOM • React compares old and new virtual DOM, only updates changed nodes

Q8: What are hooks for? Which hooks do you know? • Use state and React features in functional components • Reuse stateful logic without HOCs or render props • Common: useState, useEffect, useContext, useRef, useMemo, useCallback, useReducer • Rules: Only call at top level, only in React functions • Split code by concern, not lifecycle methods

Q9: What is the difference between useMemo and useCallback? • useMemo returns memoized VALUE, useCallback returns memoized FUNCTION • useMemo for expensive calculations, useCallback for stable function references • useCallback(fn, deps) is the same as useMemo(() => fn, deps) • Use when passing to memoized children or as dependencies • Don't overuse - adds memory overhead, profile first

Q10: What's the difference between useEffect and useLayoutEffect? • useLayoutEffect runs synchronously before paint, useEffect after paint

Q11: What are the three ways to use useEffect? • No dependency array: runs after every render • Empty array []: runs once on mount, cleanup on unmount • With dependencies [count]: runs when dependencies change • Always include cleanup function for subscriptions/listeners • Runs AFTER render (asynchronous), not during

Q12: How do you create a custom hook? • Function name must start with "use" • Can use other hooks inside • Returns values/functions for component to use • Example: useLocalStorage, useDebounce, useToggle • Benefits: code reuse, separation of concerns, easier testing

Q13: What are React keys and why are they important? • Help React identify which items changed, added, or removed in lists • Should be stable, unique identifiers (not array index)

Q14: What is prop drilling and how to avoid it? • Passing props through multiple component levels • Intermediate components don't use props, just pass them down • Avoid with: Context API, state management (Redux/Zustand), component composition • Prop drilling is OK for 2-3 levels • Choose solution based on app size and state complexity

### STATE MANAGEMENT QUESTIONS

Q15: What are three principles of Redux? • Single source of truth: entire app state in one store • State is read-only: only way to change is dispatch action • Changes made with pure functions: reducers are pure functions • Makes debugging easier, enables time-travel • Applies to most state managers (core principles)

Q16: What are reducers? • Pure function: (state, action) => newState • Takes current state and action, returns new state • No side effects, no mutations, predictable output • Example: case 'INCREMENT': return { ...state, count: state.count + 1 } • Redux Toolkit uses Immer to simplify immutable updates

 
Q17: What is middleware in Redux? • Sits between dispatching action and reaching reducer • Intercepts actions for side effects (API calls, logging, etc.) • Function signature: store => next => action => { } • Common middleware: redux-thunk (async), redux-saga, RTK Query • Example uses: logging, crash reporting, async API calls, routing

### ARCHITECTURE & PATTERNS QUESTIONS

Q18: Explain code splitting and lazy loading in React applications. • Split bundle into smaller chunks loaded on demand • React.lazy(() => import('./Component')) for component-level splitting • Suspense component shows fallback while loading • Route-based splitting most common (load page when navigated) • Reduces initial bundle size, faster first paint

Q19: Explain React Context performance issues and how to solve them. • All consumers re-render when context value changes • Problem: even if component uses only part of context • Solutions: split contexts by concern, useMemo for context value • Use state management for frequently changing data • Or use context selectors (use-context-selector library) 

### REAL-WORLD SCENARIO QUESTIONS

Q20: How would you handle authentication and protected routes in React? • Store auth state in Context or Redux • Create ProtectedRoute wrapper checking auth status • Redirect to login if not authenticated • Store token in httpOnly cookie or localStorage (security tradeoff) • Refresh token strategy for long sessions • Check auth on app mount and route changes

Q21: How would you handle rendering 10,000+ items in a table or list? • Virtualization: Only render visible items + small buffer (react-virtuoso, react-window) • Pagination: Split into pages, load 50-100 items at a time • Infinite scroll: Load more as user scrolls (Intersection Observer) • Backend pagination: Don't fetch all data at once, use limit/offset or cursor • For tables: virtualized tables with fixed row height for best performance • Avoid: rendering all 10,000 DOM nodes (causes lag, memory issues)

Q22: Explain CORS. Why do we get CORS errors and how do you fix them? • Cross-Origin Resource Security: browser blocks requests to different origin • Origin = protocol + domain + port (http://localhost:3000 vs http://localhost:5000) • Browser sends preflight OPTIONS request for certain requests • Server must respond with Access-Control-Allow-Origin header • Fix: configure backend CORS headers, use proxy in development (Vite proxy) • In this project: vite.config.ts can proxy API requests to avoid CORS in dev

Q23: A request times out or fails. How would you handle it and what retry strategies exist? • Show error message to user with retry button • Automatic retry with exponential backoff (wait 1s, 2s, 4s between retries) • Implement timeout handling with AbortController • For real-time updates: short polling (request every X seconds) vs long polling (keep connection open) • Short polling: simple but inefficient, long polling: better but complex • Modern approach: WebSocket for persistent connection, fallback to polling • RTK Query has built-in retry logic and timeout configuration

 Q24: Backend isn't ready yet. How would you develop the frontend? • Use MSW (Mock Service Worker) to intercept network requests • Define mock handlers with realistic data structure and delays • Same mocks work in development browser and tests • Agree on API contract (endpoints, request/response shapes) with backend team • Use TypeScript interfaces for API types (shared contract) • Can switch from mocks to real API by changing one flag • Benefits: develop independently, test edge cases (errors, slow responses), demos without backend
